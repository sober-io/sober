# Conversations

A conversation is the primary unit of interaction with Sõber. Each conversation has its own message history, settings, tool context, and (optionally) a workspace directory.

---

## Creating a Conversation

From the web UI, click **New Conversation** in the sidebar. You can also create conversations via the REST API:

```http
POST /api/v1/conversations
Content-Type: application/json

{
  "title": "Rust project setup"
}
```

The response includes the conversation ID, which is used in all subsequent API calls and WebSocket messages.

A conversation title is auto-generated by the agent from the first few messages if you do not supply one.

---

## Group Conversations

Conversations can be scoped to a group, allowing multiple users to participate. Group conversations share a group-scoped memory context in addition to each participant's user-scoped memory. To create a group conversation, supply a `group_id` at creation time.

Group membership and permissions are managed through the auth system. Group admins can invite members and control who has access to group-scoped knowledge.

---

## Conversation Settings

Each conversation carries settings that influence how the agent responds:

| Setting | Description | Default |
|---------|-------------|---------|
| Model | The LLM model used for this conversation. | System default (from `[llm].model` in config). |
| Temperature | Sampling temperature (0.0 = deterministic, 1.0 = creative). | Provider default. |
| Max tokens | Maximum tokens per response turn. | From `[llm].max_tokens` in config. |

Settings can be adjusted per-conversation through the UI or the API.

## Sending Messages

Messages are sent via HTTP POST and acknowledged immediately with a stored message ID. The agent processes the message asynchronously and emits events through the subscription stream.

```http
POST /api/v1/conversations/{id}/messages
Content-Type: application/json

{
  "content": "Explain Rust's ownership model",
  "role": "user"
}
```

The response is a lightweight ack — the actual agent reply arrives through the WebSocket.

## WebSocket Connection

The frontend maintains a single persistent WebSocket connection to `/api/v1/ws`. This connection receives events for all conversations the user is subscribed to.

### Connection lifecycle

1. Connect to `ws://<host>/api/v1/ws` with a valid session cookie or `Authorization: Bearer <token>` header.
2. The server subscribes the connection to the user's active conversations.
3. Events arrive as JSON frames for the lifetime of the connection.
4. The client reconnects automatically with exponential backoff on disconnect.

### Subscribing to a conversation

Send a subscription message after connecting:

```json
{ "type": "subscribe", "conversation_id": "<uuid>" }
```

Unsubscribe when navigating away:

```json
{ "type": "unsubscribe", "conversation_id": "<uuid>" }
```

### Server-sent event types

All events carry `conversation_id` and a typed `event` payload:

| Event type | Description |
|------------|-------------|
| `NewMessage` | A complete new message was stored (role, content, message ID). |
| `TextDelta` | A streaming text chunk from the current agent response. |
| `ThinkingDelta` | A reasoning/thinking chunk (extended thinking models only). |
| `TitleChanged` | The conversation title was updated (by auto-generation or the user). |
| `ToolCallStart` | The agent is about to call a tool (name, input). |
| `ToolCallResult` | A tool call completed (name, output, is_error). |
| `ConfirmRequest` | The agent needs user confirmation before running a command (confirm_id, command, risk_level). |
| `Done` | The current agent turn completed. |
| `Error` | An error occurred during processing. |

### Confirmation flow

When the `shell` tool classifies a command as requiring confirmation and the workspace's permission mode is `PolicyBased` or `Interactive`, a `ConfirmRequest` event is sent. Permission mode is a workspace setting managed via `PATCH /conversations/{id}/settings`. The frontend shows a confirmation dialog. The user's response is sent back:

```json
{ "type": "confirm", "confirm_id": "<uuid>", "approved": true }
```

The agent then re-executes the command with `confirmed: true`.

### Keepalive

The client sends a ping every 30 seconds:

```json
{ "type": "ping" }
```

The server responds with `{ "type": "pong" }`. Connections that miss too many pings are closed by the server.

## Message Types

Messages have a `role` field:

| Role | Description |
|------|-------------|
| `user` | Message from the human user. |
| `assistant` | Message from Sõber. |
| `tool` | Tool call result (internal, not displayed directly). |
| `system` | System-level context (injected by the agent, not shown in UI). |

---

## Conversation History

The full message history for a conversation is available via:

```http
GET /api/v1/conversations/{id}/messages
```

Pagination is supported with `limit` and `before` query parameters.

---

## Deleting a Conversation

```http
DELETE /api/v1/conversations/{id}
```

This archives the conversation. The workspace directory and artifacts associated with the conversation are retained; they are not deleted automatically.
